\section{Ollama}%
\label{sec:Ollama}

Ollama\footnote{\href{https://ollama.com/}{https://ollama.com/}}, van het gelijknamig bedrijf, is een gratis open-source software dat het gebruik van LLMs vereenvoudigt. Deze software maakt het mogelijk om LLMs te downloaden op een lokaal machine zodat ze ook offline kunnen draaien. De software  behandelt de achterliggende complexiteit grotendeels zelf af, dus deze moet eerst geïnstalleerd zijn om met de gedownloade modellen te kunnen werken.\\

\subsection{RAM Vereisten van Offline LLMs}
\label{ssec:ramvereistenofflinellms}

Bij de cloud modellen is het niet zo zeer van belang hoe groot deze zijn want ze zullen het werkgeheugen van dat ander systeem belasten. Bij het gebruiken van offline modellen is de hoeveelheid beschikbare RAM wel een sterk limiterende factor voor het kiezen van een LLM. \\

Bijvoorbeeld heeft gpt-oss:120b 80GB RAM nodig om vlot te werken, volgens de release informatie van \autocite{OpenAI2025}:

\begin{quote}
    "The weights for both gpt-oss-120b and gpt-oss-20b are freely available for download on Hugging Face and come natively quantized in MXFP4. This allows for the gpt-oss-120B model to run within 80GB of memory, while gpt-oss-20b only requires 16GB."
\end{quote}

Het was niet mogelijk om een 20b(gpt-oss:20b) model te laten draaien op computer A. Door middel van Quantization \todo{link to reference} neemt gpt-oss:20b(20b $\rightarrow$ 20 miljard) parameter model 16GB RAM in beslag.  Computer A gebruikt al wat RAM om andere achtergrondprocessen te laten draaien en er blijft nadien niet meer genoeg om modellen van deze grootte in te laden. \\

Figuur \ref{fig:ollamaram} toont een piek van RAM gebruikt bij het lokaal draaien van qwen3-vl:8b op Computer A. Ollama zorgt ervoor dat de machine nooit meer dan de beschikbare RAM zal gebruiken bij het runnen van een lokale LLM, in Ollama-CLI zal er een foutboodschap getoond worden dat aangeeft hoeveel het tekort aan resources bedraagt. Dit is ook hoe er geweten is dat model gpt-oss:20b niet kan draaien op Computer A. Ook bij het gebruik van de Ollama App, of Ollama Code zal het runnen van de LLM onderbroken worden als er andere processen zorgen voor een cumulatief teveel aan RAM-gebruik. Wanneer een LLM aan het runnen is blijft die gedurende 5 minuten ingeladen als achtergrond proces(ollama ps). Wanneer tijdens deze periode een ander model opgestart wordt. Ollama ondersteunt twee lopende LLMs tegelijkertijd\footnote{\href{https://github.com/ollama/ollama/blob/main/docs/faq.mdx}{https://github.com/ollama/ollama/blob/main/docs/faq.mdx}}, wanneer het systeem genoeg beschikbare werk geheugen heeft zal de nieuwe LLM ook opgestart worden, anders zal het eerder opgestarte model afgesloten worden door Ollama (ollama stop <modelnaam>) om genoeg RAM vrij te maken voor de nieuwe opstart.

\begin{itemize}

\end{itemize}

In het geval van de Lokale LLM wordt een sha256 file (dat gigabytes groot kan zijn) bewaard, bij de cloud modellen worden die op LLM-naam > LLM-versie bewaard en dit zijn manifest files die zo ~1kb plaats innemen.

\subsubsection{Ollama API}
\label{sssec:ollamaapi}

Ollama heeft een gereserveerd poortnummer(11434) waarmee verbonden kan worden gebruikmakend van TCP(Transmission Control Protocol)\footnote{\href{https://en.wikipedia.org/wiki/List\_of\_TCP\_and\_UDP\_port\_numbers}{https://en.wikipedia.org/wiki/List\_of\_TCP\_and\_UDP\_port\_numbers}} met de bijschrift: "Ollama to run LLM locally". Deze API kan gebruikt worden zonder internet verbinding.

De Ollama API endpoints\footnote{\href{https://github.com/ollama/ollama/blob/main/docs/faq.mdx}{https://github.com/ollama/ollama/blob/main/docs/faq.mdx}} die aangeroepen kunnen worden met API-requests zijn:
\begin{itemize}
    \item \textbf{http://localhost:11434/api/generate}: Generate wordt gebruikt om een bepaald LLM een eenmalige prompt te sturen.
    \item \textbf{http://localhost:11434/api/chat}: Chat wordt gebruikt om meerdere vragen(prompts) na elkaar te kunnen stellen aan een bepaald LLM.
\end{itemize}

Deze API zal verschillende keren gebruikt worden bij \ref{ssec:ollamametcode}.
Hiervoor zullen we wat code nodig hebben van \autocite{OllamaApi}.

\subsubsection{Ollama CLI \& Commandos}
\label{sssec:ollamacliandcommandos}

De CLI kan aangesproken worden vanuit een Windows cmd omgeving of powershell omgeving. Het is ook aangeraden om in de CLI een lokaal LLM te downloaden omdat de download-progressie ook wordt afgebeeld. Hier volgen een aantal nuttige commando's\footnote{\href{https://docs.ollama.com/cli}{https://docs.ollama.com/cli}} om via de CLI LLMs(lokaal en cloud) te gebruiken. \\

\begin{itemize}
    \item \textbf{ollama --version}: Hiermee controleren we de installatie van de Ollama software
    \item \textbf{ollama pull <modelnaam>}: Downloadt een LLM.
    \item \textbf{ollama run <modelnaam>}: Start een LLM op, waarna je automatisch in een chatvenster zit om ermee te communiceren. Als de LLM nog niet gedownload is zal deze commando de LLM downloaden.
    \item \textbf{/bye}: Deze input zal de chatvenster met de LLM verlaten, men kan ook "ctrl + d" in typen om hetzelfde te bekomen.
    \item \textbf{ctrl + c}: Onderbreken van een LLM-output.
    \item \textbf{ollama stop <modelnaam>}: Stopt een draaiende LLM.
    \item \textbf{ollama ls}: Geeft een lijst weer van geïnstalleerde LLMs (of cloud-manifesten) en hoeveel ruimte ze op de machine gebruiken.
    \item \textbf{ollama ps}: Toont de actieve LLM proces(sen).
    \item \textbf{ollama show <modelnaam>}: Geeft wat info over een bepaald model (architectuur, parameters, context lengte, embedding lengte, quantizatie methode, temperatuur(0|1), top\_k, top\_p)
    \item \textbf{"""}: Start/einde van een multi-line input.
     \item \textbf{ollama rm <modelnaam>}: Verwijdert een LLM van de harde schijf.
     \item \textbf{ollama signin}: Hiermee wordt een link uitgeprint, wanneer deze gevolgd wordt zal de computer verbonden worden met de Ollama account, deze device is nu gekoppeld aan de account. Dit is een manier om cloud modellen te mogen gebruiken(een andere manier is met een API-key).
     \item \textbf{ollama signout}: De computer loskoppelen van de Ollama account.
\end{itemize}

Om een \textbf{afbeelding in ollama CLI} te laten analyseren door een vision model gebruikt men een commando dat deze structuur heeft:

\begin{quote}
    \textit{ollama run qwen3-vl:8b-instruct "what's in this image? \textbackslash Users\textbackslash peter\textbackslash Voorbeeld 01.jpg"}
\end{quote}

Ollama CLI kan om met afbeeldingen, achter de schermen wordt deze geconverteerd naar iets dat de LLM kan interpreteren, een base64 geëncodeerde string. Er moet enkel een absoluut pad meegegeven worden naar de afbeelding vanuit de root van de PC, dus alles wat achter C: staat.

\subsection{Ollama met Code \& Tijds-Controle}

Nu dat we zeker drie manieren kennen om te communiceren met Ollama LLMs is er een vergelijkende test gedaan op vlak van efficiëntie, gemeten in snelheid van response in seconden bij de verschillende manieren van werken met de LLMs. Tussen elke test werd de Bash commando 'Ollama stop <llm-naam>' gebruikt zodat de opstart tijd van de LLM ook meegerekend werd. 
Bij onderstaande code fragmenten wordt er vermeld als ze offline of online moeten gebruikt worden. Wanneer het online is betekent dit dat deze methode van communiceren met de LLM een internet verbinding nodig heeft. Voor deze tests werd python versie 3.12.9 gebruikt. \\

Een test met de \textit{qwen3-vl:8b-instruct} model, dit is een variant van de \textit{qwen3-vl:8b} LLM die het denk proces niet zal weergeven in de output. Dat tonen van het denk proces vertraagt namelijk de response tijd aanzienlijk en in sommige gevallen, zoals bij moeilijkere vragen of een lange chatsessie kan de LLM vastlopen. \\

Eerst werd de Ollama\footnote{\href{https://ollama.com/download}{https://ollama.com/download}} software op Computer A gedownload. \\

Om de LLM te gebruiken moest deze eerst op de lokale machine Computer A aanwezig zijn. Dit werd gedownload naar Computer A met volgende Bash commando:
\textit{ollama pull qwen3-vl:8b-instruct}

Het downloaden van een LLM kan wat tijd in beslag nemen, dit zijn volledige LLMs, in dit voorbeeld \textit{qwen3-vl:8b-instruct} neemt 6.1GB schijfruimte in. \\

\section{Omgaan met API Keys \& Secrets}
\label{sec:omgaanmetapikeysensecrets}
Tot nu toe werd nogal kort door de bocht gegaan met de API-sleutels. In echte projecten is het de bedoeling dat mensen niet zomaar aan een API-sleutels kunnen. Er kan best zorgvuldig mee omgesprongen worden om te vermijden dat derden toegang krijgen tot je sleutels, deze zijn al kort besproken, zie \ref{ssec:apikeys}. Als deze sleutels openbaar worden gemaakt kan iemand anders de service gebruiken zonder dat je het weet. Door de account info te controleren en verdacht gebruik vast te stellen kan men concluderen dat er een API-sleutel gelekt is. Gelukkig zijn deze gemakkelijk om aan te maken en kan er op elk moment bij twijfel een nieuw sleutel gebruikt worden. \\

Men zorg best dat API-keys nooit op code repositories terecht komen door ze te bewaren in een .env file of een andere locatie dat in de .gitignore file zit en dus niet mee op de (mogelijk) publiek beschikbare repository terecht komt. \\

De oplossing dat gebruikt werd in de komende demo's en code-fragmenten is het bewaren van de sleutels in een folder dat in de .gitignore zit en in de python demo's wordt die sleutel met een functie binnen gehaald.  \\

Buiten een project heeft men soms geen .env file en zijn er maar losse files en folders, om in python zelf een functie vanuit een andere file te importeren moet een python file eerst als module gezien worden door je IDE. Dit kan verwezenlijkt worden door een leeg \_\_init\_\_.py file aan te maken in de folder die python files bevat waaruit functies geëxporteerd moeten worden. Nu kunnen alle functies in de folder met die init file geïmporteerd worden in de python demo's. \\

\begin{verbatim}

We gebruiken opnieuw Ollama's API om te verbinden met een gpt-oss variant, dit wordt ook in de Settings ingesteld. Binnen de main functie gebruiken we een prompt dat een hint geeft dat we op zoek zijn naar een deel van dit onderzoek-document, deze zal bij het retrieval proces gematched worden met de relevante content uit de databron. \\

De volgende 3 lijnen zijn een representatie van hoe LlamaIndex het RAG proces simplificeert. \\

\begin{enumerate}
    \item \textbf{SimpleDirectoryReader:} De variabele 'documents' wijst naar onze opgehaalde lijst van documenten die met SimpleDirectoryReader allen zijn ingeladen, klaar voor de volgende stap.
    \item \textbf{VectorStoreIndex:} De variabele 'index' zijn de geïndexeerde documenten, VectorStoreIndex handelt gelijk de document-chunking, embedding en vector-store af. De methode .from\_documents() krijgt een lijst van Document objecten mee en heeft ook een transformations parameter waaraan men een ingestion pipeline kan meegeven, dat kan TextSplitters en NodeParsers bevatten voor meer controle over hoe de documenten worden geïndexeerd. Hier kan trouwens ook een meer persistente vector-DB aan worden meegegeven, op dit moment wordt het enkel in-memory bijgehouden.
    \item \textbf{query engine:} LlamaIndex' query engine handelt al het volgende achter de schemren af: prompt embedding, retrieval, prompt augmentation, LLM querying en response generation. Dit vereenvoudigt de het RAG proces aanzienlijk! Dit zet onze VectorStoreIndex om naar een query engine, het is ook mogelijk om deze naar een retriever om te zetten met \textit{index.as\_retriever()}.
\end{enumerate}

De functie search\_documents is een tool die de krachtige query\_engine kan aanspreken om de relevante documenten(of delen ervan) op te halen, dit geeft de augmented prompt terug aan de agent die hier nu een heel context-onderbouwd antwoord aan de gebruiker kan geven. De functie bevat ook interne documentatie-string dat de LLM signaleert dat dit een handige tool is om te gebruiken wanneer er gevraagd wordt om info uit een document te halen dit kan een sterke invloed hebben voor het al dan niet aanroepen van de functie. \\

De agent wordt aangemaakt, deze is van het type AgentWorkFlow en kan trouwens meerdere agents bevatten, deze krijgt de zoek tool mee, de gebruikte LLM wordt ingesteld en een systeem prompt dat wellicht bekend voorkomt. De uiteindelijke response wordt gegenereerd als returnwaarde van de agent.run() methode, deze methode heeft veel meer mogelijke argumenten, onder anderen: user\_msg / chat\_history / memory / ctx / max\_iterations / early\_stopping\_method / start\_event. Deze worden voorlopig niet besproken. \\

\subsection{LlamaIndex vs LangChain}

LlamaIndex heeft krachtige, hoge level (wrapper-) klassen en simplificeert verschillende stappen in het RAG proces. \\

LangChain daarentegen heeft als voordeel dat het granulair en flexibel (en customizable) is juist omdat er meer manuele setup vereist is.

LangGraph, als natuurlijke opvolger van LangChain, kan mogelijk een nog betere oplossing bieden. Het wordt momenteel alleen onderzocht indien er voldoende tijd is, maar het verdient zeker vermelding als een veelbesproken framework.

\textbf{Uitleg van de code:} Deze is heel beknopt en gebruiksvriendelijk, men instantieert met de Interface() constructor een gelijknamig object waarop de launch() methode kan aangeroepen worden. Vervolgens ontleden we dit object:
\begin{itemize}
    \item \textbf{fn:} De functie dat men kan meegeven aan de Gratio interface. Een belangrijk onderdeel, dit zorgt ervoor dat we onze eigen code-implementaties kunnen gebruiken, deze zijn mogelijks ook heel complex in uitwerking, deze functie kan als toegangspunt dienen voor een complexe en genest systeem, dit is het belangrijkste onderdeel van deze Interface object en heeft interacties met alle volgende attributen.
    \item \textbf{inputs:} Dit voorziet allerlei verschillende soorten typische web-interface invoer van verschillende soorten data, zelfs files zijn mogelijk. Dit is een verzameling van Gradio objecten dat kan omgaan met de ingevoerde data. Deze worden uiteindelijk meegegeven aan de functie als argumenten. We zien 6 verschillende Gradio input types, dus onze functie zal 6 argumenten meekrijgen en het is aan ons de keuze hoe deze data zal verwerkt worden, in dit geval is het een eenvoudige uitwerking, maar de functie zou ook een LLM kunnen gebruiken. Er wordt een return-waarde verwacht.
    \item \textbf{outputs:} Hier kan men de return-waarde van de functie verwerken, er worden evenveel output componenten verwacht als aantal returnwaarden van de functie, wanneer men met LLMs werkt is het normaal om tekst of markdown als type output te gebruiken.
    \item \textbf{examples:} Er kan een lijst van voorbeelden klaargezet worden, wanneer de gebruiker deze aanklikt zullen de inputvelden hiermee opgevuld worden, het is dus ook de bedoeling dat het type data overeen komt met de verwachte input data types.
\end{itemize}

Er werd een system prompt ingesteld in de hoop dat de response van de AI altijd in een copieerbare maarkdown blok wordt gezet, dit zal de LLM zelf proberen afhandelen, men kan aan de AI zelfs vragen om bepaalde opmaak-code te gebruiken in zijn response. \\

Er is een cull\_state functie dat ervoor zorgt dat de chatgeschiedenis telkens afgekapt wordt om tegelijk te zorgen dat de LLM iets van verleden chat-context heeft en een maximum in te stellen van context, rate limiting dus. In essentie zorgt deze functie dat de system\_prompt blijft bestaan en de memory max 20 berichten lang blijft. Opvallend nu is dat we yield gebruiken wat in python een generator-functie maakt, ideaal om een iteratief-antwoord stapsgewijs terug te sturen. De functie moet altijd 2 returnwaarden hebben omdat de Gradio Interface dit verwacht. De systeem bericht wordt eenmalig aan de state toegevoegd daarna de gebruiker prompt. \\

De gekozen LLM, een letterlijke string-waarde dat gekoppeld is aan de radio-select en mee in de argumenten van de functie zit wordt nu via de Ollama client chat functie als argument meegegeven, alsook de state wat de chatgeschiedenis bevat. Dit is trouwens een heel manueel aanpak om de chatgeschiedenis te beheren en heeft soms negatieve gevolgen, vermoedelijk leid dit tot hallucinaties, dit werd niet verder onderzocht, maar er bestaan hogere level functies die gemaakt zijn om de chatgeschiedenis te beheren, dat even terzijde. De opgebouwde stream van tekst wordt teruggestuurd met yield, samen met de state. Op het einde van de functie wordt de opgebouwde stream in zijn geheel toegevoegd als assistant bericht aan de state en wordt er met cul\_state de oudste berichten verwijdert(als er meer dan 20 zouden zijn). \subsection{Offline vs Deployed}

Kan ook wel als dev. vs prod. omschreven worden. Er moeten enkele kleine aanpassingen gemaakt worden om een lokaal(offline) werkend demo online uit te rollen op HF-spaces.

\textbf{API-keys:} Bij de offline werkende demo's werden de API-keys geïmporteerd uit een aparte folder en in de os-environment als variabele opgeslagen, bij online projecten moet men voor een bepaalde space naar de settings > variables and secrets > secrets de API-sleutel opslaan met als sleutel de exact verwachte naam, vb.: "OLLAMA\_API\_KEY" of "TAVILY\_API\_KEY" en waarde is de waarde van de API-key(zonder ""), deze worden dan opgeslagen in de environment van die space. Wanneer het nodig is om deze expliciet in de code te gebruiken kan men binnen het online project deze code toevoegen:

\begin{minted}{python}
    import os
    
    TAVILY_API_KEY = os.getenv("TAVILY_API_KEY")
\end{minted}

\textbf{Installaties:} Om bepaalde code te gebruiken werd regelmatig een pip install gedaan in combinatie met een import. Dit moet nu anders gebeuren, op de repository moet er nu een .txt file zijn genaamd 'requirements.txt'. Hier in staan alle packages(dependencies) die nodig zijn om het project te laten lopen. Wanneer hier bijvoorbeeld ollama in staat zal binnen die online omgeving \textit{pip install ollama} gebeuren. Zo voegt men gescheiden met een enter, of \verb|\n| de nodige packages aan de file toe die geïnstalleerd moeten worden. \\

\textbf{Deployment trigger:} Dit is redelijk gemakkelijk te voorzien, zoals 'index' vaak verwacht wordt als naam voor de root van een project wordt op HF-spaces 'app' verwacht, dus bij aanmaak of wijziging van de app.py file zal een build + deploy getriggerd worden van die space. \\

Het moet vermeld worden dat koppelen van VSC met HF-space voor versiecontrole en commit + push niet zomaar werkte, maar men kan makkelijk de project files uploaden op de space en alsnog kan de build + deployment worden getriggerd. \\

Volgens de installatie gids \footnote{\href{https://docs.crewai.com/en/installation}{https://docs.crewai.com/en/installation}} kan met de volgende commando's een project boilerplate gegenereerd worden dat men al kan gebruiken om AI-projecten verder uit te werken.\\

Installeer eerst de py package manager \textbf{uv}. BeeAI guides raden deze trouwens ook aan. Uv beheert Python-versies en packages in geïsoleerde virtuele omgevingen, zodat je projectafhankelijkheden niet globaal op je systeem hoeven te worden geïnstalleerd en conflicten tussen projecten worden voorkomen. We zullen niet alle commando's bekijken, uiteindelijk voert men deze code uit in de terminal op het project te bouwen: \textbf{crewai create crew crewai\_demo\_project}. Waarna een aantal vragen gesteld worden:\\

\textbf{Select a provider to set up:} \\
We kunnen kiezen uit:
\begin{multicols}{3}
    \begin{enumerate}
        \item OpenAI       \item Anthropic    \item Gemini
        \item NVIDIA NIM   \item Groq         \item HuggingFace
        \item Ollama       \item Watson       \item Bedrock
        \item Azure        \item Cerebras     \item SambaNova
        \item Other
    \end{enumerate}
\end{multicols}

\textbf{Select a model to use for Ollama:} \\
Keuzes waren:
\begin{enumerate}
    \item ollama/llama3.1
    \item ollama/mixtral
\end{enumerate}

Na het kiezen van Ollama als provider en llama3.1 als model moest dit model enkel nog gedownload worden(\textbf{ollama pull llama3.1}), en een bestaand kant en klare demo kon al draaien(\textbf{crewai run}), dit is zeker een zinvolle werkwijze wanneer men grootschalige AI-projecten wil maken. Wat er ook bij zit maar niet te zien is in de file-structuur is een \textbf{AGENTS.md} file van meer dan 1000 lijnen instructies bedoeld voor AI-agents, best practices, algemene regels, dependencies, guardrails, human in the loop en veel meer. \\

\section{Vergelijking Conclusies}
\label{sec:vergelijkingconclusies}
% many different frameworks all kind of boil down to: provide a wrapper for an LLM, using bespoke design LLM-agent management systems 

De bestaande AI- en agentic frameworks evolueren allemaal en integreren de nieuwste technologische ontwikkelingen binnen AI. Sommigen liepen voorop met multi-agent functionaliteiten, de rest sluit nu aan en velen vereenvoudigen de code via abstractielagen en wrapper-klassen. Hoewel er kleine verschillen bestaan tussen de frameworks, zijn er meer gelijkenissen dan verschillen, elk framework streeft ernaar zoveel mogelijk features aan te bieden zodat gebruikers toegang hebben tot de volledige AI-toolkit: MCP \& toolgebruik, RAG, A2A en multi-agent workflows.
