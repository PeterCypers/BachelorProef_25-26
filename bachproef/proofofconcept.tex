\chapter{Proof of Concept}%
\label{ch:proofofconcept}

Hoewel RAG kan ingezet worden voor specifieke taken zoals het opvragen van informatie uit een grote interne bron van kennis zijn er andere, meer zinvolle technieken die met ondersteuning van AI gebruikt kunnen worden om APIs te helpen ontwikkelen, testen en documenteren. \\

RAG kan alsnog van pas komen voor het interpreteren en vragen stellen over de officiële OAS documentatie, als alternatief kan men ook grote documenten meegeven aan AI agents om zo zeker te zijn dat alle context beschikbaar blijft. Hier moet er telkens een afweging gemaakt worden of het inladen van documenten wel of niet voor een teveel aan context zorgt en dus verhoogde kosten van token-gebruik en een grotere kans op hallucinaties als nevenwerking teweegbrengt, maar dan kan de complexiteit van het opzetten van een RAG systeem vermeden worden, zoals besproken in Sectie: RAG vs Long Context (\ref{ssec:ragvslongcontext}). \\

\section{RAG Voor vragen over OAS}
\label{sec:ragvoorvraagenoveroas}

Door een bron van kennis op te slaan in een vector database en RAG in combinatie met semantic search technieken toe te passen kan men vragen stellen over bijvoorbeeld een verzameling van documenten, of één groot document met een minimale kost van tokens en tijd. Dit kan mogelijks ingezet worden om info op te vragen over de eigen APIs of andere APIs en zo AI assistentie krijgen voor het ontwikkelen van eigen API(s). Zie Sectie: LlamaIndex RAG \& OAS ( \ref{ssec:llamaindexragandoas}) voor uitgewerkte demo's. Het is opmerkelijk sneller en genereert betere antwoorden wanneer dit persistent in een vector database (of andere opslagmedium) wordt opgeslagen en eerst getransformeerd met toegevoegde metadata in de vorm van titels en voorbeeld- vragen/-antwoorden. Mogelijks kunnen officiële documenten zoals de OpenAPI Specification al tot de trainings-data van grote LLMs behoren. Zie Sectie: Ethiek Rond Data Harvesting (\ref{sec:ethiekronddataharvesting}).


\section{Scramble als basis voor API Documentatie}
\label{sec:scramblealsbasisvoorapidocumentatie}
% single point of truth
% commands, brief talk about, and refer to the section where I put the images and resources

Er bestaan tools om automatisch een OAD, OpenAPI document te genereren binnen een project, één zo'n tool is Scramble, deze kan ingezet worden om aan de hand van bestaande files waarin API-endpoints gedefinieerd zijn deze info extraheren en een openapi.json of openapi.yaml file genereren conform het officiële Open API Specification (OAS). Deze actie kan herhaaldelijk uitgevoerd worden om de bestaande file te updaten, zo bestaat er een \textbf{Single Point Of Truth} (SPOT) als API documentatie. Dit kan ook gebruikt worden om routes die in andere files gedefinieerd zijn in een file op te slaan, de input file en output filenaam kunnen geconfigureerd worden, zie ook Sectie: Scramble (\ref{sssec:scramble}).

Postman gebruikt Chai\footnote{\href{https://www.chaijs.com/}{https://www.chaijs.com/}}'s behavior-driven development (BDD) syntax om leesbare tests te schrijven \autocite{Postman}. De structuur van de testfuncties is een beetje gelijkaardig aan dat van Pest. \\


\textbf{Postman test syntax:}
\begin{minted}{php}
    pm.test("Status test", function () {
        pm.response.to.have.status(200);
    });
\end{minted}

\textbf{Pest (Expectations) test syntax:}
\begin{minted}{php}
    test('sum', function () {
        $value = sum(1, 2);
    
        expect($value)
            ->toBeInt()
            ->toBe(3)
            ->not->toBeString()
            ->not->toBe(4);
    });
\end{minted}

\subsection{POC Conclusies}
\label{ssec:pocconclusies}
