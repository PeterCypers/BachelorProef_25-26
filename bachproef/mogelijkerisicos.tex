\section{Security \& Privacy}
\label{sec:securityandprivacy}
%https://github.blog/news-insights/company-news/updates-to-github-copilot-interaction-data-usage-policy/
% \autocite{Rodriguez2026}

Er is een serieus gevaar dat met de sterke algemene kennis waarover LLMs beschikken er alsmaar sneller security vulnerabilities worden gevonden, zoals zero-day exploits. Met de opkomst van Claude Mythos werd de AI-security innitiatief Project Glasswing gelanceerd \autocite{Anthropic}. Dit nieuw LLM heeft al verscheidene kwetsbaarheden ontdekt in software dat al jarenlang op de markt is, het wordt almaar duidelijker dat cybersecurity experts voor grote uitdagingen zullen komen te staan in de nabije toekomst. Anthropic zal dit nieuwste en sterkste Claude model niet direct openbaar maken voor het wijde publiek, maar wil dit eerst met een beperkte groep software bedrijven delen en zelf ook verdedigingen opbouwen om zich klaar te stellen tegen de mogelijke onvoorziene aanvals capaciteiten van dit nieuw model. Deze Claude Mythos dat heel sterk scoort op codeer-benchmarks zou heel sterk kunnen zijn bij zowel de aanvallende als verdedigend kant van cybersecurity. \\

Er is een legitiem zorg bij tech-bedrijven dat hun eigen code zal gebruikt worden om AI te trainen, in vele gevallen is deze zorg gerechtvaardigd, initieel werd AI getrained op grote hoeveelheden internet scraped data en GitHub zelf heeft eigen AI-coding agent GitHub Copilot getraind op publieke open-source repositories \autocite{Claburn2022}. In een recente aankondiging deelt Github ook mee dat de code waarmee GitHub Copilot werkt zal gebruikt worden om op te training tenzij gebruikers een 'opt-out' doen, dat wil zeggen de feature afzetten, maar het zal standaard aan staan \autocite{Rodriguez2026}. Dit is ook het geval met OpenAI's ChatGPT, deze zal standaard gebruiker input gebruiken als toekomstig AI-trainingsdata, tenzij de feature wordt afgezet('opt-out') door de gebruiker. \\

Er wordt nu wel een onderscheid gemaakt tussen normale, pro en enterprise abonnementen, op enterprise code zal niet getraind worden, volgens github \autocite{Rodriguez2026}. \\

We zijn in een overgangsperiode tussen een tijdperk waar LLMs  misschien konden helpen met het vinden van vulnerabilities en een tijdperk waar men met een 1-shot prompt zwaktes zal kunnen ontdekken in systemen die al goed onderzocht zijn en versterkt tegen aanvallen. Big-tech bedrijven waarschuwen ons voor dit feit.

\label{ssec:aiaanvalsoppervlak}

De aanvalsoppervlak van AI systemen bevat alle manieren dat deze kan uitgebuit worden, op vlak van data, infrastructuur, applicaties en gebruikers \autocite{Ramnarayanan2025}.\\

AI kan gevoelige trainings-data en secrets lekken. Ongeoorloofd gebruik van AI kan voor security problemen zorgen. APIs kunnen het systeem kwetsbaar maken voor prompt-injection \autocite{Ramnarayanan2025}. \\


\subsection{Don't Trust, Verify}
% don't just accept AI-output, always check if it's correct
LLMs hebben een groote verzameling van kennis en zeker bij het coderen kunnen ze helpen bij het invullen van 'knowledge gaps' maar men kan soms heel wat tijd verliezen met debuggen wanneer ze code produceren dat mogelijks kan werken maar niet toepasselijk is binnen het huidige systeem, verouderd of niet performant is. Het inzetten van AI voor productie van boilerplate, herhalen van repetitieve structuren, het helpen met gevonden bugs of review van code. Maar bevestig telkens door vergelijking met de officiele doccumentatie, gelijkaardige implementaties in dezelfde code-base, feedback van senior developers dat de AI-aanbevolen code implementatie klopt. \\

\subsubsection{Shit goes in, Shit comes out}
Kwaliteit van gegenereerde output heeft een direct verband met de data waarop het model is getraind, als we ervan uitgaan dat een normaal distributie van betrouwbaarheid van de verzamelde data, of kwaliteit van code waarop is getraind kunnen we ervan uitgaan dat er gemiddeld of lager dan gemiddelde kwaliteit kan gegenereerd worden. De verwachtingen waren hoog bij de voorlopers van AI, waar men er namelijk van uit ging dat AI-systemen vooral expertenkennis zou bevatten \autocite{ComputerChronicles1984}.

\subsection{Rate Limiting}

Er moet een zekere overzicht en controle blijven op het gebruik van LLMs op de input en output token-gebruik, mogelijks moet men hier bepaalde limieten op zetten om de kosten voor het gebruik van de zekere service te beperken, of zeker niet de controle te verliezen. Net zoals code-efficiëntie (vb.: lussen binnen lussen) kan het voorvallen dat er gelijkaardige problemen zich voor doen bij het werken met LLMs. Hoe vaak wordt de context meegestuurd? Hoe veel tokens is dit telkens per prompt binnen een bepaalde sessie. Hebben de gebruikte tools/context gebruik en internet zoekfunctionaliteiten een heel groot aantal tokens nodig. De verscheidene kosten per input-/output token kunnen verschillen afhankelijk van de gekozen provider. Bij het gebruiken van LLMs in code is er een paramater, namelijk: "context\_window" waarmee er een limiet kan gezet worden op de hoeveelheid context dat er gebruikt wordt, zie onderstaande code fragment: \ref{lst:rate_limiting_code}.


\begin{listing}
    \begin{minted}{python}
        agent = FunctionAgent(
            tools=[multiply],
            llm=Ollama(
                model="qwen3-vl:8b-instruct",
                request_timeout=360.0,
                # Manually set the context window to limit memory usage
                context_window=8000,
            ),
            system_prompt="You are a helpful assistant that can multiply two numbers.",
        )
    \end{minted}
    \caption[Rate Limiting voor LLMs]{Zelf de maximum context aanpassen, in essentie rate-limiting van LLM-tokengebruik.}
    \label{lst:rate_limiting_code}
\end{listing}

Verschillende soorten rate limiting kunnen ingesteld worden afhankelijk van AI-provider, vaak heel uitgebreid controle is mogelijk, we zien bijvoorbeeld met Claude\footnote{\href{https://platform.claude.com/docs/en/api/rate-limits}{https://platform.claude.com/docs/en/api/rate-limits}} AI RPM(Maximum Requests per Minuut); ITPM(Maximum Input Tokens per Minuut); OTPM(Maximum Output Tokens per Minuut). Prompt caching is mogelijk zodat voor gelijkaardige prompts de gecachete data gebruikt wordt en dus minder tokens nodig zijn.
